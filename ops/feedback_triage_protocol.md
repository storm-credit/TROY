# feedback triage protocol

## 1. purpose

- 이 문서는 외부 피드백이 들어온 뒤 오케스트라가 그것을 어떻게 분류하고 처리할지 잠근다
- 목표:
  - 인상 비평, 취향 차이, 실제 수정 신호를 섞지 않고 다룬다
  - 원고와 submission stack을 불필요하게 흔들지 않는다

## 2. truth source

- `ops/alpha_reader_feedback_intake_opening.md`
- `ops/submission_release_readiness_audit.md`

## 3. classification

- keep:
  - 피드백을 읽었지만 현재 구조를 유지하는 편이 더 맞는 경우
- adjust:
  - 작은 wording, 안내문, 순서, 경미한 presentation 수정이 유효한 경우
- hold:
  - 흥미로운 의견이지만 한 명의 취향일 가능성이 높아 보류하는 경우
- investigate:
  - 실제로 여러 레이어에 영향을 줄 수 있어 근거 확인이 필요한 경우
- canon-risk:
  - 수정하면 작품의 핵심 감정 문법이나 이미 잠근 캐논을 흔들 위험이 있는 경우

## 4. preferred handling rule

- 단일 의견 하나만으로 원고 본문을 크게 바꾸지 않는다
- 같은 지적이 반복될 때만 구조 수정 후보로 올린다
- submission surface 문제는 빠르게 고칠 수 있지만, manuscript 문제는 더 엄격하게 본다
- `좋다 / 싫다`보다 `어디서 어떻게 읽혔는지`가 있는 의견을 우선한다

## 5. escalation rule

- 아래 중 하나면 오케스트라 재심으로 올린다
  - 초반 진입 장벽이 반복적으로 지적된다
  - 관계 속도나 결말 톤에 같은 문제가 여러 번 들어온다
  - motif 이해 실패가 반복된다
  - showcase selection이 실제 반응과 맞지 않는다

## 6. director note

- 피드백 triage의 목적은 많이 고치는 것이 아니라, 무엇을 지킬지 더 정확히 아는 것이다
- 오케스트라는 `수정`보다 먼저 `유지할 이유`를 확인해야 한다
